# 5. 生命周期与状态

## 5.1 状态设计原则

系统状态分为两类：

1. 用户展示状态：用于桌面客户端和管理端展示。
2. 系统内部状态：用于服务端任务处理、校验、系统预审、审核和审计。

普通用户不得直接看到复杂内部状态。页面必须以清晰、少量、稳定的状态描述业务结果。

## 5.2 扩展展示状态

扩展在社区和管理端展示以下状态：

| 状态 | 说明 | 是否进入社区搜索 | 是否进入榜单 | 是否可打开详情 | 是否可安装或接入 |
|---|---|:---:|:---:|:---:|:---:|
| 已发布 | 当前版本已通过审核并上架 | 按可见选项 | 按可见选项 | 按可见选项 | 按授权范围 |
| 已下架 | 不再对新用户提供发现、安装或接入 | 否 | 否 | 管理端可看，作者可从提交记录查看 | 否，存量可按规则继续使用 |
| 已安全下架 | 因安全风险停止提供新使用，并提示存量用户 | 否 | 否 | 管理端可看，作者可从提交记录查看 | 否，存量使用按安全策略处理 |
| 已归档 | 终态，不再作为活跃扩展使用 | 否 | 否 | 管理端可看 | 否 |

规则：

- 已发布扩展默认对登录用户展示卡片和详情。
- 勾选“仅授权范围内展示”的扩展只对授权范围内用户、作者和有管理权限的管理员展示，并且不进入榜单统计。
- 授权范围只影响主操作是否可用，不替代可见选项。
- 已下架、已安全下架和已归档扩展不在普通社区展示，但必须保留历史记录、审核记录、审计记录和必要的作者追踪入口。

## 5.3 版本状态

扩展版本状态包括：

| 状态 | 说明 |
|---|---|
| 待审核 | 版本提交后等待系统预审或管理员审核 |
| 已发布 | 当前版本已通过审核 |
| 已拒绝 | 版本申请被拒绝 |
| 已废弃 | 旧版本不再作为当前版本，但保留历史记录 |

规则：

- 同一扩展只有一个当前发布版本。
- 历史版本用于展示和审计，不允许普通用户选择安装旧版本。
- 新版本审核期间，旧版本继续可用。
- 新版本发布后，用户可收到更新提示。

## 5.4 发布申请用户展示状态

发布者在桌面客户端只看到以下申请状态：

| 状态 | 说明 | 发布者可执行操作 |
|---|---|---|
| 待审核 | 已提交，等待系统预审或管理员处理 | 撤回 |
| 需修改 | 管理员退回，要求发布者修改 | 修改后重新提交、撤回 |
| 已拒绝 | 管理员拒绝，本次申请结束 | 查看原因、复制为新申请 |
| 已通过 | 已发布或变更已生效 | 查看详情 |
| 已撤回 | 发布者主动撤回 | 复制为新申请 |

## 5.5 发布申请内部状态

服务端可使用以下内部状态：

| 内部状态 | 说明 |
|---|---|
| created | 申请已创建 |
| validating | 系统正在校验包结构、元数据、权限字段和可见选项 |
| ai_prechecking | 系统正在进行 AI 预审；AI 不可用时按系统设置降级 |
| pending_review | 等待管理员审核 |
| in_review | 管理员正在处理 |
| changes_requested | 管理员要求修改 |
| rejected | 管理员拒绝 |
| approved | 管理员通过 |
| withdrawn | 发布者撤回 |

内部状态映射到用户展示状态：

| 内部状态 | 用户展示状态 |
|---|---|
| created / validating / ai_prechecking / pending_review / in_review | 待审核 |
| changes_requested | 需修改 |
| rejected | 已拒绝 |
| approved | 已通过 |
| withdrawn | 已撤回 |

## 5.6 审核状态

管理端审核列表展示：

| 状态 | 说明 |
|---|---|
| 待处理 | 管理员可领取或直接打开处理 |
| 处理中 | 已有管理员打开并正在处理 |
| 已处理 | 已通过、已退回或已拒绝 |

要求：

- 不强制复杂锁单机制。
- 同一申请同时被多人处理时，提交审核决定必须以服务端最新状态为准。
- 若状态已变化，后提交者收到明确提示。
- 服务端必须记录实际审核决定人。

## 5.7 授权范围变更状态

扩展授权范围变更分为：

1. 授权扩大。
2. 授权收缩。

### 授权扩大

以下情况属于授权扩大：

- 本部门可用扩大为本部门及下级部门可用。
- 本部门或本部门及下级部门可用扩大为指定部门可用。
- 指定部门可用新增更多部门。
- 任意部门范围扩大为全员可用。

规则：

- 授权扩大必须创建发布申请。
- 需要部门管理员或系统管理员审核。
- 审核期间原授权范围继续生效。
- 授权扩大不得由客户端直接生效。

### 授权收缩

以下情况属于授权收缩：

- 全员可用缩小为部门范围。
- 指定多个部门减少为更少部门。
- 本部门及下级部门可用缩小为本部门可用。

规则：

- 发布者可提交授权收缩。
- 管理员也可执行授权收缩。
- 授权收缩可直接生效，但必须记录审计日志。
- 存量已安装或已接入用户可继续保留当前本地状态，除非扩展处于已安全下架状态并被系统安全策略要求处理。
- 授权收缩后禁止超出新授权范围的用户新增安装、接入、复制配置、新增启用、更新和下载。
- 授权收缩不禁止用户停用、卸载、移除托管配置、清除本地敏感变量或清除 manual-download 本地记录。
- 服务端不能保证实时阻断离线客户端或目标工具直接加载的本地内容；客户端下一次同步后必须刷新授权收缩提示。

## 5.8 可见选项变更状态

可见选项变更分为：

| 变更 | 影响 | 是否需要审核 |
|---|---|---:|
| 默认展示 -> 仅授权范围内展示 | 无权限用户不可见，不进入榜单 | 否，但必须审计 |
| 仅授权范围内展示 -> 默认展示 | 更多用户可看到卡片和详情 | 是 |

规则：

- 可见选项不得绕过授权范围，授权范围仍是主操作权限的权威来源。
- 默认展示状态下，未授权用户可查看详情但不可执行主操作。
- 仅授权范围内展示状态下，未授权用户不能通过搜索、榜单或直接链接看到详情。
- 可见选项变化必须保存变更前后摘要。

## 5.9 扩展治理动作权限表

| 动作 | 作者 | 部门管理员 | 系统管理员 | 是否需要审核 | 审计要求 |
|---|---|---|---|---|---|
| 首次发布 | 提交申请 | 处理管理范围内申请 | 处理跨部门、全员和异常申请 | 是 | 提交和处理均写审计 |
| 发布新版本 | 提交申请 | 处理管理范围内申请 | 处理跨部门、全员和异常申请 | 是 | 提交和处理均写审计 |
| 元信息修改 | 提交申请 | 处理管理范围内申请 | 处理全局申请 | 是 | 提交和处理均写审计 |
| 授权扩大 | 提交申请 | 处理未跨出管理范围的申请 | 处理跨部门和全员申请 | 是 | 提交和处理均写审计 |
| 授权收缩 | 可提交并直接生效，或提交给管理员处理 | 可直接处理管理范围内扩展 | 可直接处理全局扩展 | 否 | 必须写审计 |
| 展示扩大 | 提交申请 | 处理管理范围内申请 | 处理跨部门、全员和异常申请 | 是 | 必须写审计 |
| 展示收缩 | 可直接生效 | 可直接处理管理范围内扩展 | 可直接处理全局扩展 | 否 | 必须写审计 |
| 普通下架 | 可下架自己发布的扩展 | 可下架管理范围内扩展 | 可下架任意扩展 | 否 | 必须写审计 |
| 安全下架 | 不可执行 | 不可执行 | 可执行 | 否 | 必须写审计并通知存量用户 |
| 重新上架 | 提交重新上架申请 | 可处理管理范围内非安全下架扩展 | 可处理任意扩展，安全下架仅系统管理员可处理 | 视下架原因 | 必须写审计 |
| 归档 | 可提交归档申请 | 可归档管理范围内扩展 | 可归档任意扩展 | 否 | 必须写审计 |
| 转移维护人或归属部门 | 不可执行 | 可在管理范围内执行 | 可全局执行 | 否 | 必须写审计 |

## 5.10 下架状态

下架分为普通下架和安全下架。

### 普通下架

普通下架表示扩展不再在社区展示，也不再允许新安装、接入、下载或更新。

规则：

- 发布者可普通下架自己发布的扩展。
- 部门管理员可普通下架管理范围内扩展。
- 系统管理员可普通下架任何扩展。
- 下架必须填写原因。
- 下架原因必须区分维护、过期、违规、其他。
- 下架不删除历史版本和审计记录。
- 已安装或已接入用户不自动删除本地副本或托管配置，可继续使用当前本地状态。
- 下架后可重新上架，但必须遵守重新上架规则。

### 安全下架

安全下架表示扩展存在安全风险，需要停止新的使用并提示存量用户。

规则：

- 只有系统管理员可以执行安全下架。
- 必须填写安全原因和处置建议。
- 社区不展示，不允许新安装、接入、复制配置、下载或更新。
- 已安装或已接入用户收到强提醒。
- 是否阻断存量继续使用由系统安全策略控制；默认不强行删除用户本地文件，但本地页必须标记“安全风险”。
- 安全下架后存量策略支持三档：
  - 仅提醒：本地页标记安全风险，用户可自行停用或卸载。
  - 阻止新增启用/接入：已启用或已接入内容保留，但不得新增目标、重新接入、复制配置或更新。
  - 强制停用托管项：客户端下一次在线同步后自动停用或移除 Enterprise Agent Hub 托管的 Skill 启用、MCP 配置项或 Plugin 配置，保留本地备份和审计事件。
- 对目标工具已经直接加载或客户端离线期间运行的内容，系统只能在客户端下一次启动、在线同步且具备本地权限时处理，不能承诺立即阻断。
- 安全下架后的重新上架只能由系统管理员处理。

## 5.11 重新上架状态

重新上架表示已下架扩展恢复为已发布状态。

规则：

- 重新上架必须填写原因。
- 因维护、误操作、过期原因下架的扩展，若版本、元信息、授权范围和可见选项未变化，可由具备管理权限的管理员直接重新上架。
- 发布者申请重新上架时，按当前授权范围和可见选项计算审核归属。
- 因安全风险下架的扩展，必须由系统管理员处理。
- 因安全风险下架后重新上架，必须提供修复说明；若包内容、启动方式、权限声明、安装方式、授权范围或可见选项有变化，必须提交审核。
- 重新上架必须写审计日志。

## 5.12 归档状态

归档表示扩展进入终态。

规则：

- 归档后不得重新上架。
- 归档后不再允许新版本发布。
- 归档后不再允许授权范围或可见选项变更。
- 归档后保留历史版本、审核记录、审计记录。
- 归档必须二次确认并填写原因。
- 归档后不得再次使用相同 Extension ID 创建新扩展，历史 ID 永久保留用于追踪、审计和本地事件关联。

## 5.13 客户端本地状态

本地扩展状态包括：

| 状态 | 说明 |
|---|---|
| 未安装 / 未接入 | 本机没有该扩展的托管副本、配置或安装记录 |
| 已安装 | 已下载到 Central Store、插件目录或本机受控存储，但未启用 |
| 已接入 | MCP Server 已写入至少一个目标工具配置 |
| 已启用 | Skill 或 Plugin 已启用到至少一个工具或项目 |
| 有更新 | 服务端存在新版本 |
| 授权收缩 | 本地可继续保留和清理，但不可更新、重新安装或新增接入 |
| 已下架 | 服务端已下架，存量本地状态保留 |
| 安全风险 | 服务端安全下架或系统标记风险，需用户处理 |
| 异常 | 本地路径、Hash、配置、连接、启用位置或权限存在问题 |
| 部分成功 | 本地变更只有部分目标成功，需用户查看失败位置 |

## 5.14 状态展示要求

- 卡片只展示一个主状态。
- 详情页可展示更多状态说明。
- 复杂原因放到详情或错误提示中。
- 不使用多个状态标签堆满卡片。
- 所有状态文案必须一致，不同页面不得使用不同叫法。

## 5.15 组织快照与当前权限判断规则

组织快照用于审计和历史展示，当前权限判断必须使用服务端实时组织状态。

规则：

- 发布、审核、授权和可见选项变更必须保存提交时快照、审核通过快照和操作人快照。
- 审计展示使用快照，避免用户改名、调部门或部门改名后破坏历史记录。
- 审核归属使用提交时组织快照计算初始归属，并在管理员打开详情和提交审核决定时按当前管理员权限二次校验。
- 用户是否可安装、下载、复制配置、新增启用、配置写入或更新，使用当前用户状态、当前部门状态、当前部门归属、当前部门树和当前授权范围规则判断。
- 授权范围中保存被授权部门 ID 与 include_children 规则，部门名称快照仅用于展示和审计。
- include_children 表示按当前部门树包含该部门当前有效下级部门；新增下级部门会继承授权，停用部门不再具备有效授权。
- 部门暂不支持移动；因此不需要处理部门移动导致授权范围扩大或收缩的问题。
- 部门被停用后，该部门用户无法登录；归属部门为该部门的活跃扩展在社区视为失效，不再展示，不允许新安装、接入、复制配置、下载或更新，历史记录和审计保留。
- 部门恢复启用后，用户登录和扩展可用性按恢复后的当前状态重新计算。
- 扩展维护人或归属部门转移后，只影响后续治理权限和通知，不修改历史作者、历史部门和历史审核快照。
