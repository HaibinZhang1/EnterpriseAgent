# Skill 发布、审核、启用流程优化点

本次记录来自 2026-06-06 的本地端到端用户路径演练：桌面端发布 Skill、Chrome 管理端审核、桌面端本地启用。

## 流程证据

- 测试 Skill：`flow-skill-demo-e2e-20260606-1146`
- 发布申请：`907a0dba-b498-4d6a-97cf-1353eb221f51`
- 审核版本：`3c371092-e320-4b69-8ee9-64a9f31e854f`
- 包 SHA-256：`67cb3a5014d2f36379b1b075111f6344daa186be1f74a8df5f80254930528a97`
- 发布状态：`PUBLISHED`
- 本地启用目标：`.omx/ultragoal/enable-target`
- 本地启用结果：UI 显示 `ensure-target` 和 `link-skill` 均为 `success`，`local_targets` 记录为 `enabled`。

## 优化点清单

### P1 桌面端发布上传被阻断但错误信息过泛

- 阶段：桌面端发布向导，确认提交。
- 现象：选择合规包后点击提交，UI 报 `提交失败 参数校验失败 requestId: renderer_uploadPackage_1780717726203_79`。
- 证据：同一个顶层包含 `SKILL.md` 的 zip，通过后端 `/api/uploads/package` API 上传成功，返回 `precheck.status=PASSED` 和相同 SHA。
- 影响：用户无法从桌面端完成发布，只能靠 API 绕行；错误也没有说明是 multipart、参数绑定、包结构还是其他问题。
- 建议：保留并展示后端 `details`；为 Electron 主进程上传增加回归测试；在失败面板显示 uploadType、文件名、包结构校验结果和可复制 requestId。

### P2 设备绑定策略对账号切换场景缺少说明

- 阶段：桌面端登录切换发布者。
- 现象：退出后尝试用新普通用户登录同一桌面设备，提示 `设备已绑定到其他用户`；本次演练无法确认产品是否允许一台设备切换多账号。
- 影响：需要切换发布/审核角色的测试、客服或共享设备场景会中断，普通用户也难以判断是否应联系管理员、申请解绑或换设备。
- 建议：根据产品策略补充明确文案；若允许切换，提供解绑/重新绑定/管理员处理入口；测试环境可提供多角色设备夹具。

### P2 本地启用产物语义不够清晰

- 阶段：桌面端启用 Skill。
- 现象：启用成功后目标目录生成 symlink，指向 central-store 的版本目录；该目录内是名为 `package` 的 zip 文件，不是展开后的 `SKILL.md` 目录。
- 影响：如果“自定义目录”的产品契约是生成可直接读取的 Codex Skills 目录，当前产物还需要额外解包或适配；如果契约是链接中心仓库包，则 UI 需要解释这一点。
- 建议：明确“链接 central-store 包”与“展开为 Skill 目录”的区别；按目标场景提供链接包、展开目录或打开安装位置等模式，并在 UI 中说明后续使用方式。

### P2 包结构校验规则没有前置说明

- 阶段：桌面端上传包。
- 现象：文件夹包裹结构 `flow-skill-demo/SKILL.md` 会失败，但 UI 只显示泛化参数校验失败。
- 影响：常见 zip 打包方式容易失败，用户无法知道必须顶层直接包含 `SKILL.md`。
- 建议：上传前在客户端检查 zip 根目录；错误提示改为“Skill 包顶层必须包含 SKILL.md”；提供示例包结构。

### P2 发布向导缺少自动提取和逐步校验

- 阶段：发布向导前 3 步。
- 现象：上传包含 frontmatter 的 `SKILL.md` 后，`extensionId`、名称、简介没有自动带出；未选择包也能继续到后续步骤；错误状态在重新选择包后仍常驻。
- 影响：填写成本高，失败后用户容易误判新包仍失败。
- 建议：从 `SKILL.md` 自动提取 name/description；每步做可解释校验；文件变更后清空上一轮错误。

### P2 审核列表缺少搜索和精细筛选

- 阶段：Chrome 管理端审核。
- 现象：审核列表只有状态下拉，历史待审记录很多；本次能找到是因为最新提交排在第一行。
- 影响：积压较多时很难按 `extensionId`、提交人、时间、类型定位目标。
- 建议：增加关键词搜索、提交时间排序、提交人/类型筛选和复制申请 ID。

### P2 审核详情信息过载

- 阶段：Chrome 管理端审核详情。
- 现象：详情页直接展示大段 JSON，包括 `payloadSnapshot`、`packageSnapshot`、`storagePath` 等。
- 影响：审核者很难快速判断关键风险、变更和包内容。
- 建议：把摘要、包文件清单、风险项、权限/授权变化做结构化卡片；原始 JSON 折叠到“高级/调试信息”。

### P2 扩展详情版本信息不一致

- 阶段：桌面端社区搜索结果详情。
- 现象：搜索卡片显示 `v1.0.0 PUBLISHED`，详情页版本区域却显示 `0 个版本 / 暂无版本信息`。
- 影响：用户会怀疑审核后版本是否真正发布。
- 建议：详情页优先展示 `currentVersion`；版本接口为空时显示“当前版本 1.0.0，历史版本暂未加载”。

### P2 启用弹窗缺少可操作的路径辅助

- 阶段：桌面端启用 Skill。
- 现象：本地目标路径为空白手填，没有文件夹选择器、默认推荐目录或路径有效性预检。
- 影响：用户容易填错路径，且不知道目标目录会创建什么。
- 建议：增加文件夹选择器、最近路径、默认 Codex Skills 目录、路径权限预检和将要写入的目标预览。

### P2 启用成功后的同步状态不透明

- 阶段：启用后。
- 现象：UI 显示执行成功，但没有“查看本地目录”“跳转本地仓库”“复制目标路径”等动作；本地事件队列里 `SKILL_INSTALL`、`SKILL_ENABLE` 仍为 `pending`，UI 没有显式暴露同步完成状态。
- 影响：用户难以确认本地路径、事件同步和后续可用性之间的关系。
- 建议：成功后展示 artifact 路径、事件同步状态、下一步入口；本地仓库页支持搜索刚启用的远程 Skill。

### P3 发布确认页摘要不足

- 阶段：发布向导确认页。
- 现象：只展示名称、类型、版本、可见性、授权范围，缺少包名、SHA、风险说明、目标影响范围和校验结果。
- 影响：提交前无法有效复核。
- 建议：确认页展示可复制的 `extensionId`、包 SHA、文件清单、风险声明和授权影响。

### P3 社区结果卡片信息密度偏低

- 阶段：桌面端社区搜索结果。
- 现象：单个结果卡片占用大面积留白，关键操作只有底部“查看详情”。
- 影响：结果多时扫描效率低。
- 建议：压缩卡片高度，突出状态、版本、作者、下载/启用主按钮；保留详情作为次操作。

### P3 审核通过后的状态探针错误不易解释

- 阶段：Chrome 管理端审核通过后的状态确认。
- 现象：审核通过后，某个审核状态验证探针返回 `internal_error`，但随后通过扩展详情接口确认扩展已为 `PUBLISHED` 且 `authorized=true`。
- 影响：运维或自动化验证容易把“探针/接口选择错误”误判为审核失败。
- 建议：为审核状态查询提供明确的前端入口和稳定接口；内部错误应返回可定位的接口名、资源 ID 和下一步验证建议。

### P3 辅助功能与自动化可定位性不足

- 阶段：发布、审核、启用多处。
- 现象：管理端有多个“审核”文本需要作用域化；启用弹窗路径输入在首次快照中不稳定；下拉控件通过辅助功能接口不易切换。
- 影响：键盘用户、读屏用户和自动化回归都更容易失败。
- 建议：为关键控件增加稳定 `aria-label` / `data-testid`；所有弹窗输入框在打开时稳定暴露；下拉控件支持键盘和辅助技术选择。
