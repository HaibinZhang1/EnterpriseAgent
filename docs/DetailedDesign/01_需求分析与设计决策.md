# 01. 需求分析与设计决策

## 1.1 需求事实提炼

### 产品形态

Enterprise Agent Hub 是企业内网扩展中心，支持统一发现、发布、审核、安装和治理三类扩展：

| 类型 | 本质 | 服务端职责 | 客户端职责 |
|---|---|---|---|
| Skill | 以 `SKILL.md` 为入口的文件型能力包 | 保存包、版本、权限、审核、下载凭证、统计 | 下载、Hash 校验、Central Store、启用/停用/卸载 |
| MCP Server | MCP 接入定义，不托管业务 MCP 进程 | 保存定义清单、配置模板、变量 schema、权限、审核 | 写入目标工具 MCP 配置、保存本机敏感变量、连接检测 |
| Plugin | 面向工具的插件型扩展 | 保存包/安装清单/手动说明、权限、审核 | 通过 Tool Adapter 受控安装、写配置、手动下载记录 |

系统由三部分组成：

```text
Windows 桌面客户端：普通用户发现、安装、接入、发布申请、本地管理
服务端 API：账号、部门、权限、扩展、包、审核、审计、统计、设备、更新的权威来源
Web 管理端：管理员处理审核、治理、用户部门、审计、设备、更新、设置
```

### 运行约束

- 运行时不得依赖互联网服务。
- 服务端部署在 Linux，Docker Compose 交付。
- 数据库使用 PostgreSQL。
- 服务端文件存储卷保存扩展包、MCP 定义、Plugin 包、客户端更新包和预览文件。
- AI 系统预审只能接入企业内网 AI 服务，不可用时必须按配置降级。

### 组织与权限约束

- 只有三类角色：普通用户、部门管理员、系统管理员。
- 部门管理员管理自己部门及下级部门，且可以管理下级部门管理员。
- 服务端不得信任客户端传入的角色、部门、授权范围、可见选项判断。
- 禁止自审。
- 不允许冻结、删除最后一个可用系统管理员。
- 用户部门、角色、状态、密码变化后，既有会话必须失效。

### 审计约束

所有关键动作必须审计，并且审计日志至少包含：

```text
requestID、操作人快照、对象类型、对象 ID、动作、结果、原因、
变更前后摘要、IP、userAgent、客户端版本、设备 ID、时间
```

密码、Token、API Key、真实密钥、下载凭证明文不得写入审计和运行日志。

## 1.2 需求到设计决策映射

| 需求约束 | 设计决策 | 主要落地位置 |
|---|---|---|
| 服务端是权限和状态权威 | 建立 PermissionService、VisibilityPolicy、ScopeEvaluator；所有主操作接口二次校验 | 03、05、06 |
| 授权范围和可见选项分离 | 独立表 `extension_authorization_scopes` + `extensions.visibility_mode`；搜索先做可见过滤，操作再做授权校验 | 04、06 |
| 三类扩展共享治理但类型清晰 | `extensions` 统一主档，`mcp_definitions`/`plugin_definitions` 类型表，DTO 按类型区分 | 04、05、09 |
| 发布申请先创建 revision 再预审 | `submissions` + `submission_revisions` + `system_prechecks`，退回修改新增 revision | 04、06、10 |
| 本地写入必须可回滚 | 客户端统一 ExecutionPlan/Executor/BackupStore/LocalEventQueue | 07、08 |
| MCP 敏感变量不能上服务端 | 客户端 SecureStore 保存，服务端仅保存变量 schema 与脱敏摘要 | 07、09、11 |
| Plugin 不允许 UI 直接写目录 | UI 只能调用 IPC，IPC 调 Desktop Service，安装必须由 Tool Adapter 生成计划 | 07、08、09 |
| 离线可查看和清理本地内容 | 本地 SQLite/JSON 缓存 + 事件队列；离线只允许不依赖服务端的新主操作之外的动作 | 07、10 |
| 审计可排障 | requestID 全链路透传，AuditWriter 事务写入，本地事件携带关联 requestID | 12 |
| 内网部署 | Docker Compose、离线镜像、服务端静态资源可本地提供、AI 预审内网 Base URL | 13 |

## 1.3 关键边界

### 客户端边界

客户端可以做：

- 本机缓存；
- 本地工具扫描；
- 下载文件后的 Hash 校验；
- Central Store 安装；
- Tool Adapter 生成执行计划；
- 文件和配置事务写入；
- 本地事件队列；
- 设备登记和更新检查；
- MCP 敏感变量安全保存。

客户端不能做：

- 决定用户是否有角色权限；
- 决定授权范围是否有效；
- 决定扩展是否可见；
- 直接修改服务端扩展状态；
- 在离线状态执行新安装、新接入、新下载、新增配置写入、更新等依赖服务端授权的主操作；
- 把 MCP 密钥、API Key、下载凭证写入日志或事件摘要。

### 服务端边界

服务端可以做：

- 账号、部门、角色、会话；
- 扩展和版本权威状态；
- 授权范围和可见选项判断；
- 发布、预审、审核和治理；
- 包存储、校验、下载凭证；
- 统计、榜单、审计、通知；
- 客户端设备和更新版本；
- AI 预审编排。

服务端不能做：

- 自动部署业务 MCP Server 进程；
- 保存用户本机填写的真实 MCP 敏感变量；
- 执行上传包内脚本；
- 自动修复用户本地工具环境；
- 运行时访问互联网 AI 服务；
- 用 AI 预审替代管理员审核决定。

### Web 管理端边界

本设计只定义 Web 管理端需要调用的后端接口和数据权限，不设计前端页面布局、组件样式或交互视觉。

## 1.5 统一术语到代码命名

| 术语 | 建议代码名 | 说明 |
|---|---|---|
| 授权范围 | `AuthorizationScope` | 决定主操作权限 |
| 可见选项 | `VisibilityMode` | `PUBLIC_TO_ALL_LOGGED_IN` / `AUTHORIZED_ONLY` |
| 发布申请 | `Submission` | 首发、版本更新、元信息、授权、可见、重新上架 |
| Revision | `SubmissionRevision` | 每次提交快照 |
| 系统预审 | `SystemPrecheck` | 规则校验 + AI 预审结果 |
| 审核 | `Review` | 管理员 decision |
| 下载凭证 | `DownloadTicket` | 短期下载授权 |
| 本地事件 | `LocalEvent` | 客户端安装、写配置、回滚等事件 |
| Tool Adapter | `ToolAdapter` | 工具适配器接口 |
| 执行计划 | `ExecutionPlan` | 本地事务计划 |
| Central Store | `CentralStore` | 本地受控扩展存储 |

## 1.6 不做事项的技术约束

需求明确不做的事项必须在代码层避免预留过度复杂实现：

| 不做事项 | 代码约束 |
|---|---|
| 外部 LDAP/AD/统一身份源 | Auth 模块只实现本地手机号+密码，认证接口预留 `AuthProvider` 但不实现外部 Provider |
| macOS/Linux 桌面包 | Tool Adapter 支持平台字段保留，但默认只注册 Windows 适配器 |
| 自动脚本安全沙箱执行 | 包预审只扫描和提示风险，客户端不执行上传包内脚本 |
| 自动部署 MCP Server | MCP 模块只保存定义和配置模板，不创建服务端运行进程 |
| 多维护者协作模型 | Extension 只保留单一 maintainer，转移记录单独建表 |
| 批量安装/启用/升级 | API 和客户端服务只设计单扩展、单操作事务 |
| 自由标签 | Settings 维护固定标签，不提供用户自由标签创建 |

## 1.7 自动开发质量原则

后续代码生成时必须遵循：

1. 所有服务端写操作先鉴权，再领域校验，再状态变更，再审计。
2. 所有跨表状态生效必须有事务边界。
3. 所有接口返回稳定 `requestId` 和错误码。
4. 所有客户端文件操作必须经过执行计划，不允许在业务服务中散落 `fs.writeFile`。
5. 所有本地事件必须有幂等键。
6. 所有敏感字段必须有脱敏工具统一处理。
7. 所有 JSON snapshot 必须有版本字段，便于后续兼容。
8. 所有状态迁移必须集中在 State Machine 或 Domain Service，不允许 Controller 直接改状态。
