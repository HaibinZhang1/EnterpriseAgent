# 16. GitHub 开源参考实现

## 16.1 参考原则

本项目是企业内网扩展中心，不应直接照搬公开互联网扩展市场。参考开源项目的目的，是借鉴成熟模块边界、协议处理、安全校验和安装服务抽象。所有参考都必须适配本项目的内网、审核、授权、审计和本地事务要求。

## 16.2 modelcontextprotocol/typescript-sdk

### 项目

- 名称：`modelcontextprotocol/typescript-sdk`
- 仓库：https://github.com/modelcontextprotocol/typescript-sdk

### 参考位置

| 模块/文档 | 路径 |
|---|---|
| MCP Client Guide | `docs/client.md` |
| MCP 客户端实现 | `packages/client/src/client/client.ts` |
| Client exports | `packages/client/src/index.ts` |
| 客户端示例 | `examples/client/src/*` |
| Skill 风格文档示例 | `docs/migration-SKILL.md` |

### 借鉴点

1. **MCP transport 分类**
   `docs/client.md` 中区分 `StreamableHTTPClientTransport`、`StdioClientTransport`、`SSEClientTransport`。本项目 MCP 定义采用：
   - `remote-http` -> `streamable-http`
   - `remote-sse` -> `sse`
   - `local-command` -> `stdio`

2. **SSE legacy fallback**
   参考先尝试 Streamable HTTP，失败后 fallback 到 SSE 的思路。但本项目不在客户端自动猜测 transport，transport 由审核通过的 MCP 定义明确声明；客户端只在定义允许 legacy SSE 时使用。

3. **错误分层**
   SDK 区分协议错误和 SDK 本地错误。本项目借鉴该思想：
   - 服务端 API 错误：稳定业务错误码；
   - 本地执行错误：`DesktopError`；
   - MCP 连接检测错误：独立 `ConnectionTestResult`。

4. **请求超时和连接生命周期**
   参考 SDK 对 timeout、connection closed 的处理，本项目 MCP 连接检测必须有可配置超时，失败写本地事件，不阻塞本地配置卸载。

5. **工具/资源/提示的分页发现**
   SDK 对 listTools/listResources/listPrompts 支持分页。本项目如果未来在 Agent 能力中实际调用 MCP，可复用分页思想；当前需求只要求配置写入和连接检测，不设计 Agent 执行内核。

6. **SKILL.md frontmatter 风格**
   `docs/migration-SKILL.md` 展示了带 YAML frontmatter 的技能文档结构。本项目 Skill 包允许 `SKILL.md` 使用 frontmatter，用于提取 name、description 等候选字段，但最终发布元数据以 submission payload 为准。

### 不照搬点

- 不把 MCP SDK 作为服务端扩展市场治理模型。
- 不在服务端保存用户真实 MCP secret。
- 不让客户端根据外部注册表自动安装 MCP Server。
- 不使用互联网 OAuth 或外部 IdP 流程作为 MVP 必需项。
- 不设计 MCP tool call Agent 运行时。

### 落地设计位置

- `09_Skill_MCP_Plugin落地设计.md` 的 MCP 定义和连接检测；
- `08_本地执行事务与ToolAdapter设计.md` 的 local-command 安全；
- `14_异常处理_错误码_幂等.md` 的客户端错误分层。

## 16.3 modelcontextprotocol/registry

### 项目

- 名称：`modelcontextprotocol/registry`
- 仓库：https://github.com/modelcontextprotocol/registry

### 参考位置

| 模块/文档 | 路径 |
|---|---|
| server.json schema | `docs/reference/server-json/draft/server.schema.json` |
| API OpenAPI | `docs/reference/api/openapi.yaml` |
| schema validators | `internal/validators/schemas/*.json` |
| enhanced validation 设计 | `docs/design/proposed-enhanced-validation.md` |

### 借鉴点

1. **Manifest Schema 思路**
   registry 的 `server.schema.json` 用 JSON Schema 描述 server detail、packages、remotes、transport、inputs。本项目借鉴 schema 驱动校验方式，对 MCP 定义清单和 Plugin 安装清单建立强 schema。

2. **transport 对象化**
   参考 `streamable-http`、`sse`、`stdio` transport 建模。本项目把 accessType 和 transport 显式落库，避免只保存一段原始 JSON。

3. **secret input 标记**
   schema 中存在类似 `isSecret` 的输入标记。本项目变量 schema 使用：
   ```yaml
   sensitive: true
   type: secret
   ```
   客户端据此保存到 Windows 安全存储，服务端只保存 schema。

4. **fileSha256 完整性校验**
   registry schema 中对包文件 Hash 有明确字段。本项目所有服务端托管包和 manual-download 内网下载文件都必须登记 SHA-256，客户端下载后强校验。

5. **命令注入风险提示**
   schema 对命令行参数存在注入风险说明。本项目 MCP local-command 和 Plugin 安装清单必须禁止 shell 执行，使用 `spawn(..., shell:false)`，并将高风险命令进入审核风险提示。

6. **Repository metadata 透明度**
   registry 支持 repository metadata。本项目 MVP 不要求开源仓库，但可在未来为企业内部来源系统、制品系统、构建流水线增加 metadata，提升审核透明度。

### 不照搬点

- 不使用公共 MCP Registry 作为运行时依赖。
- 不接受互联网 registryBaseUrl 自动安装。
- 不允许外部包管理器运行时拉包。
- 不把 server.json 结构原样作为本项目 MCP 定义；本项目保留企业审核、授权、可见、风险声明字段。

### 落地设计位置

- `11_包存储_安全校验_AI预审.md` 的 schema 校验和 Hash；
- `04_服务端数据模型设计.md` 的 `mcp_definitions`；
- `09_Skill_MCP_Plugin落地设计.md` 的变量 schema 和敏感变量处理。

## 16.4 microsoft/vscode

### 项目

- 名称：`microsoft/vscode`
- 仓库：https://github.com/microsoft/vscode

### 参考位置

| 模块 | 路径 |
|---|---|
| Node 扩展管理服务 | `src/vs/platform/extensionManagement/node/extensionManagementService.ts` |
| 抽象扩展管理服务 | `src/vs/platform/extensionManagement/common/abstractExtensionManagementService.ts` |
| 扩展管理接口类型 | `src/vs/platform/extensionManagement/common/extensionManagement.ts` |
| Workbench 扩展管理服务 | `src/vs/workbench/services/extensionManagement/common/extensionManagementService.ts` |
| 扩展工作台动作 | `src/vs/workbench/contrib/extensions/browser/extensionsActions.ts` |
| 扩展同步 | `src/vs/platform/userDataSync/common/extensionsSync.ts` |

### 借鉴点

1. **扩展管理服务抽象**
   VS Code 将扩展安装、卸载、启用等动作放在 service 层，而不是 UI 层。本项目借鉴这一点，桌面 Renderer 不直接执行文件操作，必须调用客户端后端服务。

2. **本地扩展与远端 Gallery 分离**
   VS Code 有本地扩展、Gallery 扩展、工作台展示等分层。本项目对应：
   - 服务端社区扩展；
   - 客户端本地 Central Store；
   - 本地状态投影；
   - UI 只展示状态，不作为权威。

3. **安装/卸载动作状态管理**
   VS Code 的扩展动作有安装中、已安装、可更新、卸载等状态。本项目借鉴状态归一思想，但不设计 UI，只定义本地状态数据结构和后端服务。

4. **Enablement 保护**
   VS Code 有扩展启用/禁用策略。本项目借鉴“是否允许启用”独立判断，落地为 `OfflinePolicy`、`ScopeEvaluator` 和本地 `LocalExtensionState`。

5. **CLI/Workbench 分层**
   VS Code 的 CLI 和 Workbench 通过服务层复用能力。本项目的 IPC、未来命令行工具或自动化测试也应复用客户端后端服务，不复制文件操作逻辑。

### 不照搬点

- 不采用 VS Code 扩展包 VSIX 规范。
- 不实现 VS Code Marketplace。
- 不使用其扩展 Host 运行时。
- 不设计 Extension Activation 事件。
- 不把插件执行权限交给目标工具，所有托管写入仍通过 Tool Adapter。

### 落地设计位置

- `07_客户端后端架构.md` 的本地服务分层；
- `08_本地执行事务与ToolAdapter设计.md` 的执行计划；
- `09_Skill_MCP_Plugin落地设计.md` 的本地状态归一。

## 16.5 OpenAPI 与契约生成参考

虽然本次未依赖特定开源项目，建议采用以下通用实践：

- 服务端维护 OpenAPI 3.0；
- 通过 OpenAPI 生成 TypeScript API client；
- 契约测试校验响应 schema；
- 错误码作为 enum；
- 敏感字段在 schema 中显式标注 `x-sensitive: true`。

这能降低自动开发时前后端 DTO 不一致风险。

## 16.6 本项目参考实现采纳矩阵

| 设计点 | 参考项目 | 采纳方式 |
|---|---|---|
| MCP transport 建模 | modelcontextprotocol/typescript-sdk | 采用 streamable-http/sse/stdio 组合，服务端入库规范化 |

| 本地/远端状态分离 | microsoft/vscode | 服务端社区状态与客户端本地状态分离 |

## 16.7 对 Skill 参考实现的说明

公开 GitHub 中能检索到一些以 `SKILL.md` 为核心的技能示例仓库，但没有发现适合直接照搬的、同时覆盖企业内网、发布审核、授权范围、可见选项、本地事务、审计和回滚的成熟 Skill 市场实现。因此本项目 Skill 设计以需求文档定义为准，只借鉴带 frontmatter 的文档组织形式，不依赖外部 Skill registry 或 marketplace。


## 16.8 参考实现版本锁定与许可证检查

仓库存在性复核：本设计已复核 `modelcontextprotocol/typescript-sdk`、`modelcontextprotocol/registry`、`microsoft/vscode` 三个公开仓库可访问。由于本交付物不复制第三方源码，只记录参考模块和设计借鉴点；进入编码前仍必须固定具体 release tag 或 commit SHA，并在离线依赖清单中记录许可证。

为保证后续 Agent 自动开发可追溯，实际进入编码前必须为每个参考项目固定读取版本。本文档列出参考模块和借鉴点，不复制第三方代码；实现时如需要引用源码片段，必须单独做许可证评估。

| 参考项目 | 建议锁定方式 | 实现前检查项 |
|---|---|---|
| `modelcontextprotocol/typescript-sdk` | 固定 tag 或 commit SHA | transport 名称、client API 路径、许可证 |
| `modelcontextprotocol/registry` | 固定 tag 或 commit SHA | server schema 路径、OpenAPI 路径、schema license |
| `microsoft/vscode` | 固定 release tag | extensionManagement service 路径、许可证、不得复制实现代码 |
| OpenAPI 工具链 | 固定插件版本 | 生成 DTO 的 enum 命名和 nullable 规则 |

采纳限制：

1. 不把任何公开 registry 或 marketplace 作为 Enterprise Agent Hub 运行时依赖。
2. 不把外部包管理器的一键安装逻辑引入 MVP。
3. 不复制 VS Code 扩展运行时、VSIX 规范或 Marketplace API。
4. 不让 MCP SDK 的工具调用能力越过本项目审核、授权、可见选项和审计边界。
5. 如需复用开源库二进制依赖，必须在离线部署包中随版本锁定和许可证说明一起交付。
