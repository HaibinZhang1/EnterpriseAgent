# 06. 权限、可见性、审核与状态机

## 6.1 权限设计目标

权限模块必须同时满足：

1. 普通用户可使用桌面端发现和安装授权范围内扩展。
2. 部门管理员只能管理自己部门及下级部门。
3. 上级部门管理员可以管理下级部门管理员。
4. 系统管理员全局管理。
5. 服务端每次主操作重新计算授权和可见，不信任客户端。
6. 停用、卸载、本地清理不因授权收缩或下架而被禁止。

## 6.2 ActorContext

服务端所有写操作必须显式传入 `ActorContext`。

```java
record ActorContext(
    UUID userId,
    String name,
    Role role,
    UUID departmentId,
    DepartmentStatus departmentStatus,
    String sessionId,
    String deviceId,
    String clientVersion,
    String ip,
    String userAgent
) {}
```

`ActorContext` 来自服务端 session，不允许由客户端请求体传入。

## 6.4 管理范围计算

### 部门管理员范围

```text
管理范围 = 自己所属部门 + 所有当前有效下级部门
```

部门暂不支持移动，因此树关系稳定；部门停用后用户不可登录，授权判断中停用部门视为无效。

### 用户管理规则

```java
boolean canModifyUser(ActorContext actor, User target, UserPatch patch) {
    if (actor.role() == SYSTEM_ADMIN) {
        return !wouldBreakLastSystemAdmin(target, patch);
    }

    if (actor.role() != DEPARTMENT_ADMIN) return false;
    if (target.id().equals(actor.userId())) return false;
    if (target.role() == SYSTEM_ADMIN) return false;
    if (!isSelfOrDescendant(actor.departmentId(), target.departmentId())) return false;

    if (target.role() == DEPARTMENT_ADMIN) {
        if (!isStrictDescendant(actor.departmentId(), target.departmentId())) return false;
    }

    if (patch.role() == SYSTEM_ADMIN) return false;
    if (patch.departmentId() != null && !isSelfOrDescendant(actor.departmentId(), patch.departmentId())) return false;

    return true;
}
```

### 最后系统管理员保护

所有可能改变系统管理员可用性的路径都必须调用 `LastSystemAdminGuard`：

- 删除用户；
- 冻结用户；
- 修改角色；
- 修改部门导致无法登录；
- 部门停用；
- 批量导入或批量变更；
- 密码策略异常锁定不直接适用，但系统管理员被锁定时仍需保留其他可用系统管理员。

建议在事务内执行：

```sql
select id from users where role = 'SYSTEM_ADMIN' and status = 'ACTIVE' for update;
```

再计算操作后剩余可用数量。

## 6.5 授权范围模型

### Scope 类型

```java
enum ScopeType {
    DEPARTMENT,
    DEPARTMENT_TREE,
    SELECTED_DEPARTMENTS,
    ALL_EMPLOYEES
}
```

### 授权判断

```java
boolean isAuthorizedForMainOperation(User user, Extension extension, MainOperation operation) {
    if (user.status != ACTIVE) return false;
    if (user.department.status != ACTIVE) return false;
    if (extension.status != PUBLISHED) return false;
    if (extension.ownerDepartment.status != ACTIVE) return false;
    if (extension.currentVersion.status != PUBLISHED) return false;

    Scope scope = scopeRepository.findCurrent(extension.id);
    return switch (scope.type) {
        case ALL_EMPLOYEES -> true;
        case DEPARTMENT -> user.departmentId.equals(scope.singleDepartmentId());
        case DEPARTMENT_TREE -> departmentTree.isSelfOrDescendant(scope.singleDepartmentId(), user.departmentId);
        case SELECTED_DEPARTMENTS -> scope.departments.anyMatch(d ->
            d.departmentId.equals(user.departmentId) ||
            (d.includeChildren && departmentTree.isSelfOrDescendant(d.departmentId, user.departmentId))
        );
    };
}
```

注意：若授权部门被停用，则该部门用户无法登录；如果是 include_children，停用子部门也不具备有效授权。

### 主操作枚举

主操作枚举必须在权限模块、接口契约、本地事件和统计设计中保持一致。

```java
enum MainOperation {
    INSTALL,              // Skill 安装、Plugin managed-package 安装、MCP 首次接入入口
    DOWNLOAD_PACKAGE,     // Hub 托管扩展包下载
    CONTROLLED_DOWNLOAD,  // Plugin manual-download 受控下载或企业内网受控文件下载
    COPY_CONFIG,          // 授权用户复制 MCP 配置或 Plugin 配置片段
    WRITE_CONFIG,         // MCP 或 config-plugin 写入目标工具配置
    ENABLE_NEW_TARGET,    // Skill 或 Plugin 新增启用目标
    UPDATE,               // 扩展更新、MCP 模板更新、Plugin 更新
    CONNECTION_TEST       // MCP 连接检测，仅对已授权或已有托管配置的用户开放
}
```

操作映射：

| 类型 | 用户动作 | MainOperation | 是否必须在线校验 |
|---|---|---|---:|
| Skill | 安装 | INSTALL + DOWNLOAD_PACKAGE | 是 |
| Skill | 更新 | UPDATE + DOWNLOAD_PACKAGE | 是 |
| Skill | 新增启用目标 | ENABLE_NEW_TARGET | 在线优先；离线只允许最近授权缓存且无收缩/安全下架标记 |
| MCP Server | 安装到工具 / 写入配置 | INSTALL + WRITE_CONFIG | 是 |
| MCP Server | 复制配置 | COPY_CONFIG | 是 |
| MCP Server | 连接检测 | CONNECTION_TEST | 新接入需在线授权；已有托管配置可本地检测并上报事件 |
| MCP Server | 更新配置模板 | UPDATE + WRITE_CONFIG | 是 |
| Plugin managed-package | 安装 | INSTALL + DOWNLOAD_PACKAGE | 是 |
| Plugin config-plugin | 配置写入 | INSTALL + WRITE_CONFIG | 是 |
| Plugin manual-download | 受控下载 | CONTROLLED_DOWNLOAD | 是 |
| Plugin | 更新 | UPDATE | 是 |

停用、卸载、禁用、清理本地托管配置、清除本机敏感变量、清除 manual-download 本地记录不属于主操作，不因授权收缩、下架或部门停用而被禁止，但必须记录本地事件并在联网后同步。

## 6.6 可见选项模型

```java
enum VisibilityMode {
    PUBLIC_TO_ALL_LOGGED_IN,
    AUTHORIZED_ONLY
}
```

### 可见判断

```java
boolean canViewExtension(User user, Extension extension) {
    if (extension.status != PUBLISHED) return false;
    if (extension.ownerDepartment.status != ACTIVE) return false;

    if (extension.visibilityMode == PUBLIC_TO_ALL_LOGGED_IN) {
        return true;
    }

    if (hasAnyAuthorizedPrimaryOperation(user, extension)) {
        return true;
    }

    if (extension.authorId.equals(user.id) || extension.maintainerId.equals(user.id)) {
        return true;
    }

    if (user.role == SYSTEM_ADMIN) return true;

    if (user.role == DEPARTMENT_ADMIN && canManageExtension(user, extension)) {
        return true;
    }

    return false;
}

boolean hasAnyAuthorizedPrimaryOperation(User user, Extension extension) {
    return isAuthorizedForMainOperation(user, extension, MainOperation.INSTALL)
        || isAuthorizedForMainOperation(user, extension, MainOperation.DOWNLOAD_PACKAGE)
        || isAuthorizedForMainOperation(user, extension, MainOperation.COPY_CONFIG)
        || isAuthorizedForMainOperation(user, extension, MainOperation.WRITE_CONFIG)
        || isAuthorizedForMainOperation(user, extension, MainOperation.CONTROLLED_DOWNLOAD)
        || isAuthorizedForMainOperation(user, extension, MainOperation.UPDATE);
}
```

对于 `AUTHORIZED_ONLY` 未授权用户：

- 搜索不返回；
- 榜单不统计；
- 直接详情返回 `visibility_restricted`；
- 不暴露扩展名称、作者、部门、描述。

对于 `PUBLIC_TO_ALL_LOGGED_IN` 未授权用户：

- 可查看详情；
- 可 Star；
- 主操作返回不可操作原因 `scope_restricted`。

## 6.7 作者与维护人权限边界

作者和当前维护人不是额外角色。系统角色仍只有普通用户、部门管理员、系统管理员。作者 / 维护人权限是围绕具体扩展对象的对象级权限，由服务端实时判断。

| 动作 | 作者/维护人 | 是否需要审核 | 说明 |
|---|---:|---:|---|
| 查看自己扩展和提交记录 | 是 | 否 | 即使扩展 `AUTHORIZED_ONLY`，作者/维护人仍可见 |
| 撤回自己待审核或需修改申请 | 是 | 否 | 仅限 submitter |
| 退回后重新提交 revision | 是 | 重新进入审核 | 保留历史 revision |
| 普通下架自己仍可治理的扩展 | 是 | 否 | 状态变更为 DELISTED，写审计 |
| 授权收缩 | 是 | 否 | 直接生效，写审计 |
| 展示收缩 | 是 | 否 | 直接生效，写审计 |
| 发布新版本 | 是 | 是 | 创建 VERSION_UPDATE submission |
| 元信息修改 | 是 | 是 | 创建 METADATA_CHANGE submission |
| 授权扩大 | 是 | 是 | 创建 SCOPE_EXPANSION submission |
| 展示扩大 | 是 | 是 | 创建 VISIBILITY_EXPANSION submission |
| 重新上架 | 是 | 视下架原因 | 安全下架必须系统管理员处理 |
| 归档申请 | 是 | 按服务端状态判断 | 归档为终态，必须二次确认和审计 |
| 转移维护人或归属部门 | 否 | 否 | 仅管理员按管理范围执行 |

作者 / 维护人调用桌面侧 `/me/extensions/*` 接口；管理员调用 `/admin/extensions/*` 接口。两类路由可复用领域服务，但权限入口必须区分，避免普通用户依赖管理端 API。

## 6.8 授权变更分类

```java
enum ScopeChangeType {
    EXPANSION,
    REDUCTION,
    NO_EFFECT_CHANGE
}
```

### 判断原则

授权扩大：

- 部门 -> 部门树；
- 部门/部门树 -> 指定更多部门；
- 指定部门新增部门；
- 任意范围 -> 全员；
- include_children 从 false 改 true 且影响更多当前有效部门。

授权收缩：

- 全员 -> 任意部门范围；
- 指定部门减少；
- 部门树 -> 部门；
- include_children 从 true 改 false。

授权扩大必须创建 submission 并审核；授权收缩可直接生效，但必须审计。

## 6.9 可见变更分类

```java
PUBLIC_TO_ALL_LOGGED_IN -> AUTHORIZED_ONLY = VISIBILITY_REDUCTION
AUTHORIZED_ONLY -> PUBLIC_TO_ALL_LOGGED_IN = VISIBILITY_EXPANSION
```

展示收缩可直接生效并审计；展示扩大必须进入审核。

## 6.10 审核归属算法

### 输入

```java
Submission submission;
User submitter;
AuthorizationScope targetScope;
VisibilityMode targetVisibility;
SubmissionType type;
```

### 输出

```java
ReviewOwner {
  ReviewOwnerType type; // DEPARTMENT_ADMIN / SYSTEM_ADMIN
  UUID departmentId;    // 部门管理员归属，可空
}
```

### 伪代码

```java
ReviewOwner assignReviewOwner(Submission s) {
    if (s.type == VISIBILITY_EXPANSION && affectsOutsideSubmitterAdminScope(s)) {
        return systemAdmin();
    }

    if (s.targetScope.type == ALL_EMPLOYEES) {
        return systemAdmin();
    }

    Department submitterDept = s.submitterSnapshot.department;
    Optional<User> deptAdmin = findNearestEligibleDepartmentAdmin(submitterDept, excludeUser = s.submitterId);

    if (s.targetScope.type == SELECTED_DEPARTMENTS) {
        if (deptAdmin.isEmpty()) return systemAdmin();
        if (allTargetDepartmentsWithinAdminScope(deptAdmin.get(), s.targetScope)) {
            return departmentAdmin(deptAdmin.get().departmentId);
        }
        return systemAdmin();
    }

    if (deptAdmin.isPresent()) {
        return departmentAdmin(deptAdmin.get().departmentId);
    }

    return systemAdmin();
}
```

### 禁止自审

提交审核决定时：

```java
if (reviewer.id == submission.submitterId) {
    throw permission_denied("禁止自审");
}
```

系统管理员也不能审核自己提交的申请。

## 6.11 Extension 状态机

### 状态

```text
PUBLISHED
DELISTED
SECURITY_DELISTED
ARCHIVED
```

### 迁移

```text
PUBLISHED -> DELISTED             普通下架
PUBLISHED -> SECURITY_DELISTED    安全下架，仅系统管理员
DELISTED -> PUBLISHED             重新上架，视原因直接或审核
SECURITY_DELISTED -> PUBLISHED    重新上架，仅系统管理员，可能需审核
PUBLISHED|DELISTED|SECURITY_DELISTED -> ARCHIVED 归档
ARCHIVED -> *                     禁止
```

### 规则

- 下架后社区不展示，不允许新主操作。
- 安全下架后社区不展示，不允许新主操作，并通知存量用户。
- 归档为终态。
- 已归档 Extension ID 永久占用。

## 6.12 Version 状态机

```text
PENDING_REVIEW -> PUBLISHED
PENDING_REVIEW -> REJECTED
PUBLISHED -> DEPRECATED
```

版本更新审核期间旧版本继续可用。新版本发布后：

1. 新版本 `PUBLISHED`；
2. 旧版本 `DEPRECATED`；
3. `extensions.current_version_id` 指向新版本；
4. 通知已安装用户有更新。

## 6.13 Submission 状态机

内部状态：

```text
CREATED
VALIDATING
AI_PRECHECKING
PENDING_REVIEW
IN_REVIEW
CHANGES_REQUESTED
REJECTED
APPROVED
WITHDRAWN
```

用户展示映射：

| 内部状态 | 展示状态 |
|---|---|
| CREATED / VALIDATING / AI_PRECHECKING / PENDING_REVIEW / IN_REVIEW | 待审核 |
| CHANGES_REQUESTED | 需修改 |
| REJECTED | 已拒绝 |
| APPROVED | 已通过 |
| WITHDRAWN | 已撤回 |

### 迁移

```text
CREATED -> VALIDATING
VALIDATING -> AI_PRECHECKING
VALIDATING -> PENDING_REVIEW          AI 未启用
AI_PRECHECKING -> PENDING_REVIEW
PENDING_REVIEW -> IN_REVIEW           可选，不强制锁单
PENDING_REVIEW|IN_REVIEW -> APPROVED
PENDING_REVIEW|IN_REVIEW -> CHANGES_REQUESTED
PENDING_REVIEW|IN_REVIEW -> REJECTED
PENDING_REVIEW|CHANGES_REQUESTED -> WITHDRAWN
CHANGES_REQUESTED -> VALIDATING       新 revision 重新提交
```

### 并发控制

提交审核决定时使用乐观锁或状态条件更新：

```sql
update submissions
set status = 'APPROVED', updated_at = now()
where id = :id
  and current_revision_no = :revisionNo
  and status in ('PENDING_REVIEW', 'IN_REVIEW');
```

影响行数为 0 返回 `review_already_processed` 或 `submission_revision_conflict`。

## 6.14 本地状态与服务端状态关系

客户端本地状态不是服务端授权权威，但服务端通过 local events 维护投影。

| 本地状态 | 可执行动作 |
|---|---|
| 未安装/未接入 | 只能在线且授权后安装/接入 |
| 已安装 | 可离线查看、停用、卸载；新增启用需有效授权缓存且无安全下架标记 |
| 已接入 | 可停用/卸载托管 MCP 配置 |
| 有更新 | 更新必须在线重新授权 |
| 授权收缩 | 可停用/卸载/清理，不可更新/新增启用 |
| 已下架 | 可清理，不可新增主操作 |
| 安全风险 | 按安全下架策略处理 |
| 部分成功 | 必须展示失败目标并可重试/修复 |
| 异常 | 可重新扫描、修复或卸载 |

## 6.15 部门停用影响

部门停用后：

1. 该部门用户不可登录。
2. 已有会话失效。
3. 归属部门为该部门的活跃扩展在社区视为失效。
4. 不允许新安装、接入、复制配置、下载、更新。
5. 历史记录、审核记录、审计日志保留。
6. 恢复启用后按当前权限和状态重新计算。

服务端实现上，`ExtensionVisibilityService` 和 `ScopeEvaluator` 均需检查 owner department status。

## 6.18 权限设计自检点

实现时必须使用以下测试保护权限边界：

1. `AUTHORIZED_ONLY` 扩展对未授权用户搜索不可见、详情不可见、榜单不统计。
2. `PUBLIC_TO_ALL_LOGGED_IN` 扩展对未授权用户可见但主操作返回 `scope_restricted`。
3. 作者 / 维护人可见自己的扩展，但不能绕过审核执行授权扩大或展示扩大。
4. 部门管理员只能处理管理范围内对象，不能处理自己的申请。
5. 系统管理员也不能审核自己提交的申请。
6. 授权收缩、展示收缩直接生效但必须写审计。
7. 停用、卸载、清理本地托管配置不调用主操作授权判断。
8. `CONNECTION_TEST` 对新接入流程需要授权；对本机已有托管配置允许本地检测和事件上报。
