# 14. 异常处理、错误码与幂等

## 14.1 错误处理原则

1. 错误必须包含动作语义、原因和下一步修正方向。
2. 服务端使用稳定错误码，不只返回自然语言。
3. 客户端本地错误与服务端错误统一映射到 `DesktopError`。
4. 敏感信息不进入错误 message、details、日志和审计。
5. 写操作失败不得返回乐观成功。
6. 对可重试错误标记 `retryable=true`。
7. 并发状态冲突必须明确返回最新状态提示。

## 14.4 异常映射

服务端：

```java
@RestControllerAdvice
class GlobalExceptionHandler {
  @ExceptionHandler(BusinessException.class)
  ApiResponse<Void> handleBusiness(BusinessException e) {
    return ApiResponse.error(requestId(), e.errorCode(), e.message(), e.details(), e.retryable());
  }

  @ExceptionHandler(MethodArgumentNotValidException.class)
  ApiResponse<Void> handleValidation(...) {
    return ApiResponse.error(requestId(), "validation_failed", "参数校验失败", details, false);
  }

  @ExceptionHandler(Exception.class)
  ApiResponse<Void> handleUnknown(Exception e) {
    log.error("Unhandled error requestId={}", requestId(), e);
    return ApiResponse.error(requestId(), "server_unavailable", "服务暂不可用，请稍后重试", null, true);
  }
}
```

客户端：

```ts
function mapApiError(error: ApiError): DesktopError {
  return {
    code: error.code,
    message: error.message,
    details: redactor.redact(error.details),
    retryable: error.retryable
  };
}
```

## 14.5 错误 message 模板

| 场景 | message |

## 14.6 幂等策略

### 服务端写操作

使用 `Idempotency-Key`：

- 创建 submission；
- 创建 revision；
- 审核决定；
- 作者侧和管理端治理动作；
- 下载凭证签发；
- 客户端更新发布、暂停、撤回；
- 审计导出；
- 系统设置修改。

本地事件同步不使用通用业务幂等表，而使用事件自身 `device_id + idempotency_key` 去重。

幂等记录表：

```sql
idempotency_records(
  id uuid primary key,
  actor_id uuid,
  device_id varchar(128),
  idempotency_key varchar(128) not null,
  operation varchar(128) not null,
  request_hash varchar(64) not null,
  response_snapshot jsonb,
  status varchar(32) not null,
  expires_at timestamptz not null,
  created_at timestamptz not null,
  updated_at timestamptz not null
);

create unique index uk_idempotency_actor_operation_key
on idempotency_records(coalesce(actor_id, '00000000-0000-0000-0000-000000000000'::uuid), operation, idempotency_key);
```

处理逻辑：

```java
if exists same actor+key+operation:
  if request_hash same:
    return response_snapshot
  else:
    throw idempotency_conflict
else:
  execute business in transaction
  save response_snapshot
```

### 本地事件幂等

`local_events` 使用：

```sql
unique(device_id, idempotency_key)
```

重复同步返回 `IGNORED` 或原处理结果，不重复统计。

## 14.7 审核并发

审核决定必须避免重复通过/拒绝。

策略：

1. 读取 submission。
2. 校验 `status in (PENDING_REVIEW, IN_REVIEW)`。
3. 校验 `revisionId == currentRevisionId`。
4. 条件更新状态。
5. 影响行数 0 返回 `review_already_processed`。

若需要更强一致性，可使用 `select ... for update` 包住审核事务。

## 14.8 下载凭证错误

下载凭证只短期有效。

| 场景 | 错误码 |
|---|---|
| 过期 | download_ticket_expired |
| 已使用 | download_ticket_used |
| 对象不存在 | resource_not_found |
| 用户不匹配 | permission_denied |
| Hash 不匹配 | hash_mismatch |
| purpose 与对象类型不匹配 | download_purpose_invalid |
| Plugin 内网来源过期 | plugin_download_source_expired |
| 客户端更新签名非法 | signature_invalid |

服务端日志不得记录凭证明文，只记录 ticket ID 或 hash 前 8 位。

## 14.9 客户端重试策略

| 操作 | 可重试 | 策略 |
|---|---:|---|
| 搜索/详情 | 是 | 网络错误指数退避 |
| 下载 | 是 | 断点续传可选，Hash 最终校验 |
| 创建下载凭证 | 是 | 幂等键，同请求重复返回 |
| 提交发布申请 | 是 | 幂等键 |
| 审核决定 | 谨慎 | 幂等键，状态冲突提示刷新 |
| 本地事件同步 | 是 | 持久队列指数退避 |
| 本地执行计划 | 否 | 不自动重试，用户确认后重新执行 |
| 客户端更新下载 | 是 | 重试后 Hash 校验 |
| 启动安装程序 | 否 | 避免重复启动 |

## 14.10 本地回滚异常

回滚可能失败。结果必须表达：

```json
{
  "status": "PARTIAL_SUCCESS",
  "error": {
    "code": "rollback_failed",
    "message": "操作失败且部分目标回滚失败，请查看详情或联系管理员。"
  },
  "rollback": {
    "attempted": true,
    "status": "PARTIAL_SUCCESS",
    "failedItems": [
      {
        "targetMasked": "C:/Users/***/config.json",
        "reason": "文件被占用"
      }
    ]
  }
}
```

本地事件上报 `ROLLBACK_FAILED`，服务端写审计。

## 14.11 敏感异常处理

任何异常 details 若包含以下 key，应统一删除或脱敏：

```text
password
token
apiKey
secret
credential
downloadTicket
authorization
mcpVariableValue
```

路径脱敏：

```text
C:\Users\Alice\AppData\... -> C:\Users\***\AppData\...
```

URL 脱敏：

```text
https://host/path?token=abc -> https://host/path?token=***
```

## 14.12 包安全错误优先级

当一个包同时存在多个问题，返回主错误码优先级：

```text
package_path_traversal
> package_uncompressed_size_exceeded
> package_file_count_exceeded
> package_too_large
> skill_manifest_missing
> plugin_manifest_invalid / mcp_transport_invalid
> package_unsafe_file_detected
```

完整 findings 进入预审结果，但 API 主错误码只返回一个，便于客户端提示。

## 14.13 状态冲突错误

| 操作 | 冲突 | 错误码 |
|---|---|---|
| 发布新版本 | 版本未递增 | validation_failed |
| 审核决定 | 已处理 | review_already_processed |
| 重新提交 | 申请不是需修改 | validation_failed |
| 重新上架 | 已归档 | extension_archived |
| 展示扩大 | 未走审核 | validation_failed |
| 授权收缩 | 实际为扩大 | validation_failed |
| 客户端更新发布 | 版本未递增 | validation_failed |
| 撤回更新 | 已撤回 | validation_failed |

## 14.14 错误码维护规范

1. 错误码只增不随意改名。
2. 删除错误码前必须确认无客户端依赖。
3. 错误码文案可本地化，但 code 稳定。
4. 错误码文档与 OpenAPI 同步。
5. 自动测试覆盖关键错误码。


## 14.15 幂等与错误码自检用例

| 用例 | 预期 |
|---|---|
| 同一 Idempotency-Key、同一请求体重复提交发布申请 | 返回第一次的 submissionId，不创建第二条 submission |
| 同一 Idempotency-Key、不同请求体提交审核决定 | 返回 `idempotency_conflict` |
| 使用他人的 tempUploadId 创建 revision | 返回 `upload_not_owned` |
| 使用已消费 tempUploadId 创建 revision | 返回 `upload_already_consumed` |
| 下载凭证 purpose=UPDATE 但对象为客户端更新 | 返回 `download_purpose_invalid`，客户端更新必须使用 CLIENT_UPDATE |
| Plugin manual-download 来源过期 | 返回 `plugin_download_source_expired` |
| 两个管理员同时通过同一 revision | 只有一个成功，另一个返回 `review_already_processed` 或 `submission_revision_conflict` |
| PATCH 设置 expectedVersion 过旧 | 返回 `setting_version_conflict` |
