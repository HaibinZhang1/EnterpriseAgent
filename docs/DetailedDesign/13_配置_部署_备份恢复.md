## 13.1 服务端配置结构

`application.yml` 建议只放默认值，实际部署使用环境变量或挂载配置覆盖。

```yaml

  requestIdHeader: X-Request-ID

  password:
    algorithm: bcrypt
    minLength: 10
```

## 13.2 设置模块强类型映射

数据库 `settings` 中使用 key-value，但业务读取强类型对象。

| key | 类型 | 说明 |
|---|---|---|
| categories | string[] | 固定分类 |
| tags | string[] | 固定标签 |
| upload.limits | object | 上传限制 |
| password.policy | object | 密码策略 |
| session.policy | object | 会话策略 |
| audit.retention | object | 审计保留 |
| ai.precheck | object | AI 预审配置 |
| mcp.write.policy | object | MCP 写入策略 |
| plugin.install.policy | object | Plugin 受控安装策略 |
| client.update.policy | object | 客户端更新策略 |
| security.delist.policy | object | 安全下架存量策略 |

设置变更必须写 `settings_history` 和 `audit_logs`。`settings` 表必须包含 `version`，任何管理端 PATCH 都必须携带 `expectedVersion`。

设置表迁移最小 DDL：

```sql
create table settings (
  key varchar(128) primary key,
  value jsonb not null,
  version integer not null default 1,
  updated_by uuid,
  updated_at timestamptz not null default now()
);

create table settings_history (
  id uuid primary key,
  key varchar(128) not null,
  before_value jsonb,
  after_value jsonb not null,
  before_version integer,
  after_version integer not null,
  reason text not null,
  updated_by uuid not null,
  actor_snapshot jsonb not null,
  created_at timestamptz not null default now()
);
```


## 13.3 Docker Compose 结构

```yaml
version: "3.9"

services:
  api:
    image: enterprise-agent-hub-api:1.0.0
    container_name: enterprise-agent-hub-api
    ports:
      - "8080:8080"
    env_file:
      - ./config/app.env
    volumes:
      - ./data/storage:/data/enterprise-agent-hub/storage
      - ./logs/api:/logs
    depends_on:
      - postgres
    healthcheck:
      test: ["CMD", "/app/healthcheck.sh"]
      interval: 30s
      timeout: 5s
      retries: 5

  admin-web:
    image: enterprise-agent-hub-admin-web:1.0.0
    container_name: enterprise-agent-hub-admin-web
    ports:
      - "8081:80"
    depends_on:
      - api

  postgres:
    image: postgres:16
    container_name: enterprise-agent-hub-postgres
    environment:
      POSTGRES_DB: enterprise_agent_hub
      POSTGRES_USER: eah
      POSTGRES_PASSWORD: change_me
    volumes:
      - ./data/postgres:/var/lib/postgresql/data
```

可选择由 API 服务直接托管 Web 管理端静态资源，也可使用单独 Nginx 容器。无论哪种，都不得运行时从互联网加载脚本、字体、图标或统计服务。

## 13.5 Flyway 迁移

规则：

1. 每次 schema 变更必须新增 migration。
2. migration 命名：`V<major>_<minor>_<patch>__description.sql`。
3. 生产部署前备份数据库。
4. 迁移失败服务端不得进入可用状态。
5. 破坏性迁移必须在发布说明中明确。
6. 本项目全新构建，不需要旧测试版本兼容迁移。

初始迁移建议拆分：

```text
V1_0_0__identity_org.sql
V1_0_1__extensions_versions_scopes.sql
V1_0_2__submissions_reviews_prechecks.sql
V1_0_3__packages_downloads_local_events.sql
V1_0_4__audit_notifications_settings.sql
V1_0_5__devices_client_updates.sql
V1_0_6__indexes_constraints.sql
```

## 13.6 健康检查

### 服务端健康接口

```text
GET /actuator/health
```

检查项：

- API 进程；
- PostgreSQL；
- storage root 可读写；
- temp 目录可写；
- Flyway migration 状态；
- AI 预审服务可选连通性；
- 文件存储剩余空间摘要。

返回不得包含密码、Token、AI Key。

### 部署脚本 healthcheck

```bash
#!/usr/bin/env bash
set -euo pipefail
curl -fsS http://localhost:8080/actuator/health | grep -q '"status":"UP"'
```

## 13.8 backup.sh 设计

```bash
#!/usr/bin/env bash
set -euo pipefail

BACKUP_ROOT=${BACKUP_ROOT:-./backups}
TS=$(date +%Y%m%d_%H%M%S)
OUT="$BACKUP_ROOT/eah_backup_$TS"

mkdir -p "$OUT"

docker exec enterprise-agent-hub-postgres pg_dump \
  -U "$POSTGRES_USER" \
  -d "$POSTGRES_DB" \
  --format=custom \
  > "$OUT/postgres.dump"

tar -C ./data -czf "$OUT/storage.tar.gz" storage
tar -czf "$OUT/config.tar.gz" config

sha256sum "$OUT"/* > "$OUT/SHA256SUMS.txt"

echo "backup created: $OUT"
```

要求：

- 备份失败返回非零退出码。
- 输出校验摘要。
- 默认保留最近 7 次备份或按配置清理。
- 备份操作本身可写审计，若由管理端触发必须写审计。

## 13.9 restore.sh 设计

恢复流程：

1. 校验备份目录存在。
2. 校验 SHA256SUMS。
3. 确认覆盖当前数据。
4. 停止 API。
5. 恢复 PostgreSQL。
6. 恢复 storage。
7. 恢复 config。
8. 启动服务。
9. 检查 Flyway 状态。
10. 执行健康检查。
11. 检查关键包文件存在性。

伪脚本：

```bash
#!/usr/bin/env bash
set -euo pipefail

BACKUP_DIR="$1"
test -f "$BACKUP_DIR/SHA256SUMS.txt"
(cd "$BACKUP_DIR" && sha256sum -c SHA256SUMS.txt)

docker compose stop api admin-web

docker exec -i enterprise-agent-hub-postgres pg_restore \
  -U "$POSTGRES_USER" \
  -d "$POSTGRES_DB" \
  --clean --if-exists \
  < "$BACKUP_DIR/postgres.dump"

tar -C ./data -xzf "$BACKUP_DIR/storage.tar.gz"
tar -xzf "$BACKUP_DIR/config.tar.gz"

docker compose up -d
./scripts/healthcheck.sh
```

## 13.12 内网 AI 服务配置

```yaml
aiPrecheck:
  enabled: true
  baseUrl: http://ai-risk-review.internal
  model: enterprise-risk-review
  timeoutMs: 30000
  failurePolicy: CONTINUE_WITH_UNAVAILABLE
```

校验：

- baseUrl 必须是内网地址或系统管理员确认的企业地址。
- 不允许运行时配置互联网 AI endpoint。
- 修改 AI 配置必须写审计。
- 失败降级策略必须明确。

## 13.13 服务端启动顺序

```text
1. 加载配置。
2. 校验必要环境变量。
3. 检查 storage root 可写。
4. 建立数据库连接。
5. 执行 Flyway migration。
6. 初始化默认 settings。
7. 初始化默认系统管理员策略，若无管理员则进入首次初始化流程。
8. 启动 HTTP 服务。
9. 启动 Outbox、清理任务、AI 预审任务、审计保留任务。
```

首次初始化可通过命令行或受控 API 创建第一个系统管理员，完成后必须关闭初始化入口。

## 13.14 配置错误处理

| 场景 | 行为 |
|---|---|
| 数据库不可连接 | 服务端启动失败 |
| storage 不可写 | 服务端启动失败 |
| Flyway 失败 | 服务端启动失败 |
| AI 配置不可连通 | 服务端可启动，AI 健康项 WARNING |
| 分类/标签为空 | 使用默认值并记录启动日志 |
| 密码策略非法 | 启动失败或拒绝保存设置 |
| 审计保留低于 180 天 | 拒绝设置 |

## 13.15 运维交付检查

部署完成后必须验证：

1. Docker Compose up 成功。
2. `/actuator/health` 为 UP。
3. 管理端静态资源可内网访问。
4. PostgreSQL 数据持久化。
5. storage 可写。
6. 创建用户和登录成功。
7. 审计日志可查询。
8. 上传包和删除 temp 正常。
9. 客户端设备可登记。
10. 备份脚本和恢复脚本可执行。
11. 无互联网访问依赖。


## 13.16 设置版本与并发修改

系统设置表必须支持版本号或等价乐观锁，避免多个系统管理员互相覆盖配置。

建议 settings value：

```json
{
  "key": "aiPrecheck",
  "version": 3,
  "value": {
    "enabled": true,
    "baseUrl": "http://ai.internal",
    "timeoutMs": 30000,
    "failurePolicy": "CONTINUE_WITH_UNAVAILABLE"
  },
  "updatedAt": "2026-05-07T10:00:00Z"
}
```

`PATCH /admin/settings/{key}` 必须携带 `expectedVersion`。服务端发现版本不一致时返回 `setting_version_conflict`，不写成功审计。

## 13.17 离线包清单一致性检查

重新打包前必须检查：

1. `docker-compose.yml`、镜像 tar、配置 example、迁移脚本、备份恢复脚本齐全。
2. Web 管理端静态资源不引用互联网字体、脚本、图标或统计服务。
3. 桌面客户端安装包包含 SHA256SUMS 和签名摘要。
4. 服务端配置示例包含 `storage.tempTtlHours`、`audit.retentionDays`、`aiPrecheck.failurePolicy`、`clientUpdate.signatureRequired`。
5. 运行时不访问互联网；AI 服务地址必须为内网地址或未配置时降级人工审核。
