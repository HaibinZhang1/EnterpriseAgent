# 00. 阅读指南与设计结论

## 0.1 设计目标

本文档包面向后续 Agent 自动开发，目标不是复述需求，而是把 `docs/RequirementDocument` 中的产品约束转换为可执行的技术设计依据，使代码生成能够遵循清晰模块边界、稳定接口契约、确定数据结构和可验证流程。

本设计覆盖：

- Windows 桌面客户端的后端能力：Electron Main Process、本地执行层、Central Store、Tool Adapter、下载与 Hash 校验、本地事务、事件同步、设备与更新。
- 服务端：Spring Boot 3 + Java 21 模块化单体、PostgreSQL、Flyway、包存储、权限、审核、审计、AI 预审、设备、更新、部署。
- 三类扩展：Skill、MCP Server、Plugin。
- 关键流程：发布、预审、审核、安装、接入、启用、更新、卸载、下架、归档、离线同步、客户端更新。
- 可落地的接口、DTO、表结构、领域服务、错误码、幂等、异常处理、配置和测试建议。

本设计不覆盖：

- 客户端前端 UI 视觉、布局、样式。
- Web 管理端前端 UI 视觉、布局、样式。
- Agent 执行内核、模型调用、任务编排、工具调用运行时。桌面端 `Agent` 仅作为入口边界保留。

## 0.3 核心设计结论

### 结论一：服务端采用模块化单体，不做微服务

当前目标规模为几百人级别，需求强调审核生效、权限校验、审计写入和包状态一致性。微服务会引入分布式事务、跨服务审计链路、部署复杂度，不符合内网离线部署和快速落地目标。

服务端采用：

```text
Spring Boot 3 + Java 21 + PostgreSQL + Flyway
模块化单体
Controller -> ApplicationService -> DomainService -> Repository
```

所有核心写操作在一个业务事务内完成状态变更与审计记录；通知、异步 AI 预审、统计刷新通过 Outbox 或事务后事件解耦。

### 结论二：客户端后端必须独立于 UI 层

桌面 UI 不允许直接写文件、改工具配置、执行安装、保存密钥。Electron Main Process 内必须形成独立本地后端：

```text
Renderer
  -> IPC Facade
  -> Desktop Application Service
  -> Local Executor / Tool Adapter / Secure Store / Local DB / API Client
```

这样后续 Agent 自动开发时，即使前端 UI 变化，本地事务、回滚、Tool Adapter、事件同步仍保持稳定。

### 结论三：Extension 是统一治理对象，三类扩展保持类型边界

服务端使用统一 `extensions` 主表管理公共字段，使用 `extension_versions` 管理版本和包快照，使用类型表保存差异：

```text
extensions
  ├─ extension_versions
  │    ├─ mcp_definitions
  │    └─ plugin_definitions
```

Skill 不需要额外定义表，Skill 的 `SKILL.md`、README、文件清单和风险摘要进入 version metadata snapshot。

### 结论四：授权范围与可见选项必须完全分离

授权范围只控制主操作：

- 新安装；
- 新下载；
- 复制配置；
- 新增启用；
- 配置写入；
- 更新。

可见选项只控制未授权用户是否可发现和打开详情：

```text
public_to_all_logged_in：所有登录用户可浏览，未授权不可操作
authorized_only：仅授权范围内展示，无权限用户不可见、不进榜单
```

停用、卸载、本地清理不受当前授权收缩、下架或部门停用影响，但重新安装、重新接入、更新必须重新走服务端授权。

### 结论五：本地安装和配置写入必须计划化、事务化、可追踪

客户端后端必须实现统一 `ExecutionPlan`：

```text
dry-run -> 生成影响摘要 -> 执行 -> 备份 -> 原子写 -> 校验 -> 回滚或部分成功 -> 本地事件入队
```

所有 Skill 启用、MCP 配置写入、Plugin 安装/卸载都复用该执行模型，Tool Adapter 只负责生成计划和适配工具格式，不直接决定权限。

### 结论六：Submission Revision 是审核与预审的事实来源

发布申请必须先创建 `submissions` 和 `submission_revisions`，再执行规则校验和 AI 预审。退回修改时创建新 revision，不能覆盖历史 revision。审核通过时记录生效 revision。

### 结论七：审计日志与运行日志通过 requestID 串联

每次关键操作形成链路：

```text
客户端 requestID
 -> 服务端运行日志
 -> 审计日志
 -> 本地事件
 -> 客户端设备事件
 -> 管理端审计查询
```

敏感值不得进入日志、审计、本地事件摘要或 AI 预审输入。

## 0.4 推荐实现顺序

按风险和依赖建议分四期开发，每期可独立验收。

### P0：底座

1. 服务端工程、Flyway、统一响应、错误码、requestID。
2. 用户、部门、角色、登录、会话、密码策略。
3. 审计日志基础表和写入切面。
4. 设置模块、分类/标签/上传限制。
5. 客户端 API Client、deviceID、本地目录、日志、事件队列骨架。

### P1：扩展与发布审核闭环

1. Extension、Version、Scope、Visibility。
2. Package Storage、Download Ticket、包校验。
3. Submission、Revision、System Precheck、Review。
4. 管理端所需审核/扩展管理接口。
5. 通知基础能力。

### P2：客户端本地闭环

1. Central Store、Hash 校验、ExecutionPlan、Rollback。
2. Tool Adapter 接口与默认适配器注册。
3. Skill 安装、启用、更新、卸载。
4. MCP 配置模板渲染、写入、连接检测、变量安全存储。
5. Plugin managed-package/config-plugin/manual-download。

### P3：治理、统计、设备、更新与运维

1. Local Event 同步、统计、榜单。
2. 下架、安全下架、归档、重新上架、维护人/归属部门转移。
3. 客户端设备、心跳、版本分布。
4. 客户端更新包、Hash/签名校验、更新事件。
5. Docker Compose 离线部署、备份恢复脚本。

- 三类扩展落地设计补充 manual-download 受控下载、Hash 校验、内网来源过期和统计拆分。
